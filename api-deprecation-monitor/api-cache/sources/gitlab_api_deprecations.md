These deprecations often signal improved API features and recommend using new fields
or endpoints for features.</p><p>Though some deprecations mention a v5 REST API, no v5 REST API development is active.
GitLab will not make these changes within REST API v4, and <a href=/api/rest/#versioning-and-deprecations>follows semantic versioning for the REST API</a>.</p><h2 id=geo_nodes-api-endpoints><code>geo_nodes</code> API endpoints</h2><p>Breaking change. <a href=https://gitlab.com/gitlab-org/gitlab/-/issues/369140>Related issue</a>.</p><p>The <a href=/api/geo_nodes/><code>geo_nodes</code> API endpoints</a> are deprecated and are replaced by <a href=/api/geo_sites/><code>geo_sites</code></a>.
It is a part of the global change on <a href=/administration/geo/glossary/>how to refer to Geo deployments</a>.
Nodes are renamed to sites across the application. The functionality of both endpoints remains the same.</p><h2 id=merged_by-api-field><code>merged_by</code> API field</h2><p>Breaking change. <a href=https://gitlab.com/gitlab-org/gitlab/-/issues/350534>Related issue</a>.</p><p>The <code>merged_by</code> field in the <a href=/api/merge_requests/#list-merge-requests>merge request API</a>
has been deprecated in favor of the <code>merge_user</code> field which more correctly identifies who merged a merge request when
performing actions (set to auto-merge, add to merge train) other than a simple merge.</p><p>API users are encouraged to use the new <code>merge_user</code> field instead. The <code>merged_by</code> field will be removed in v5 of the GitLab REST API.</p><h2 id=merge_status-api-field><code>merge_status</code> API field</h2><p>Breaking change. <a href=https://gitlab.com/gitlab-org/gitlab/-/issues/382032>Related issue</a>.</p><p>The <code>merge_status</code> field in the <a href=/api/merge_requests/#merge-status>merge request API</a>
has been deprecated in favor of the <code>detailed_merge_status</code> field which more correctly identifies
all of the potential statuses that a merge request can be in. API users are encouraged to use the
new <code>detailed_merge_status</code> field instead. The <code>merge_status</code> field will be removed in v5 of the GitLab REST API.</p><h3 id=null-value-for-private_profile-attribute-in-user-api>Null value for <code>private_profile</code> attribute in User API</h3><p>Breaking change. <a href=https://gitlab.com/gitlab-org/gitlab/-/issues/387005>Related issue</a>.</p><p>When creating and updating users through the API, <code>null</code> was a valid value for the <code>private_profile</code> attribute, which would internally be converted to the default value. In v5 of the GitLab REST API, <code>null</code> will no longer be a valid value for this parameter, and the response will be a 400 if used. After this change, the only valid values will be <code>true</code> and <code>false</code>.</p><h2 id=single-merge-request-changes-api-endpoint>Single merge request changes API endpoint</h2><p>Breaking change. <a href=https://gitlab.com/gitlab-org/gitlab/-/issues/322117>Related issue</a>.</p><p>The endpoint to get
<a href=/api/merge_requests/#retrieve-merge-request-changes>changes from a single merge request</a>
has been deprecated in favor the
<a href=/api/merge_requests/#list-merge-request-diffs>list merge request diffs</a> endpoint.
API users are encouraged to switch to the new diffs endpoint instead.</p><p>The <code>changes from a single merge request</code> endpoint will be removed in v5 of the GitLab REST API.</p><h2 id=managed-licenses-api-endpoint>Managed Licenses API endpoint</h2><p>Breaking change. <a href=https://gitlab.com/gitlab-org/gitlab/-/issues/397067>Related issue</a>.</p><p>The endpoint to get all managed licenses for a given project has been deprecated in favor the
<a href=/user/compliance/license_approval_policies/>License Approval policy</a> feature.</p><p>Users who wish to continue to enforce approvals based on detected licenses are encouraged to create a new <a href=/user/compliance/license_approval_policies/>License Approval policy</a> instead.</p><p>The <code>managed licenses</code> endpoint will be removed in v5 of the GitLab REST API.</p><h2 id=approvers-and-approver-group-fields-in-merge-request-approval-api>Approvers and Approver Group fields in Merge Request Approval API</h2><p>Breaking change. <a href=https://gitlab.com/gitlab-org/gitlab/-/issues/353097>Related issue</a>.</p><p>The endpoint to get the configuration of approvals for a project returns
empty arrays for <code>approvers</code> and <code>approval_groups</code>.
These fields were deprecated in favor of the endpoint to
<a href=/api/merge_request_approvals/#list-all-approval-rules-for-a-merge-request>list all approval rules</a>
for a merge request. API users are encouraged to switch to this endpoint instead.</p><p>These fields will be removed from the <code>get configuration</code> endpoint in v5 of the GitLab REST API.</p><h2 id=runner-usage-of-active-replaced-by-paused>Runner usage of <code>active</code> replaced by <code>paused</code></h2><p>Breaking change. <a href=https://gitlab.com/gitlab-org/gitlab/-/issues/351109>Related issue</a>.</p><p>Occurrences of the <code>active</code> identifier in the GitLab Runner GraphQL API endpoints will be
renamed to <code>paused</code> in GitLab 16.0.</p><ul><li>In v4 of the REST API, you can use the <code>paused</code> property in place of <code>active</code></li><li>In v5 of the REST API, this change will affect endpoints taking or returning <code>active</code> property, such as:<ul><li><code>GET /runners</code></li><li><code>GET /runners/all</code></li><li><code>GET /runners/:id</code> / <code>PUT /runners/:id</code></li><li><code>PUT --form "active=false" /runners/:runner_id</code></li><li><code>GET /projects/:id/runners</code> / <code>POST /projects/:id/runners</code></li><li><code>GET /groups/:id/runners</code></li></ul></li></ul><p>The 16.0 release of GitLab Runner will start using the <code>paused</code> property when registering runners.</p><h2 id=runner-status-will-not-return-paused>Runner status will not return <code>paused</code></h2><p>Breaking change. <a href=https://gitlab.com/gitlab-org/gitlab/-/issues/344648>Related issue</a>.</p><p>In a future v5 of the REST API, the endpoints for GitLab Runner will not return <code>paused</code> or <code>active</code>.</p><p>A runner&rsquo;s status will only relate to runner contact status, such as:
<code>online</code>, <code>offline</code>, or <code>not_connected</code>. Status <code>paused</code> or <code>active</code> will no longer appear.</p><p>When checking if a runner is <code>paused</code>, API users are advised to check the boolean attribute
<code>paused</code> to be <code>true</code> instead. When checking if a runner is <code>active</code>, check if <code>paused</code> is <code>false</code>.</p><h2 id=runner-will-not-return-ip_address>Runner will not return <code>ip_address</code></h2><p>Breaking change. <a href=https://gitlab.com/gitlab-org/gitlab/-/issues/415159>Related issue</a>.</p><p>In GitLab 17.0, the <a href=/api/runners/>Runners API</a> will return <code>""</code> in place of <code>ip_address</code> for runners.
In v5 of the REST API, the field will be removed.</p><h2 id=default_branch_protection-api-field><code>default_branch_protection</code> API field</h2><p>Breaking change. <a href=https://gitlab.com/gitlab-org/gitlab/-/issues/408315>Related issue</a>.</p><p>The <code>default_branch_protection</code> field is deprecated in GitLab 17.0 for the following APIs:</p><ul><li><a href=/api/groups/#create-a-group>New group API</a>.</li><li><a href=/api/groups/#update-group-attributes>Update group API</a>.</li><li><a href=/api/settings/#update-application-settings>Application Settings API</a></li></ul><p>You should use the <code>default_branch_protection_defaults</code> field instead, which provides more finer grained control
over the default branch protections.</p><p>The <code>default_branch_protection</code> field will be removed in v5 of the GitLab REST API.</p><h2 id=require_password_to_approve-api-field><code>require_password_to_approve</code> API field</h2><p>The <code>require_password_to_approve</code> was deprecated in GitLab 16.9. Use the <code>require_reauthentication_to_approve</code> field
instead. If you supply values to both fields, the <code>require_reauthentication_to_approve</code> field takes precedence.</p><p>The <code>require_password_to_approve</code> field will be removed in v5 of the GitLab REST API.</p><h2 id=pull-mirroring-configuration-with-the-projects-api-endpoint>Pull mirroring configuration with the projects API endpoint</h2><p>Breaking change. <a href=https://gitlab.com/gitlab-org/gitlab/-/issues/494294>Related issue</a>.</p><p>In GitLab 17.6, the <a href=/api/project_pull_mirroring/#update-pull-mirroring-for-a-project-deprecated>pull mirroring configuration with the Projects API</a> is deprecated.
It is replaced by a new configuration and endpoint, <a href=/api/project_pull_mirroring/#update-project-pull-mirroring-settings><code>projects/:id/mirror/pull</code></a>.</p><p>The previous configuration using the Projects API will be removed in v5 of the GitLab REST API.</p><h2 id=restrict_user_defined_variables-parameter-with-the-projects-api-endpoint><code>restrict_user_defined_variables</code> parameter with the projects API endpoint</h2><p>In GitLab 17.7, the <a href=/api/projects/#update-a-project><code>restrict_user_defined_variables</code> parameter in Projects API</a>
is deprecated in favour of using only <code>ci_pipeline_variables_minimum_override_role</code>.</p><p>To match the same behavior of <code>restrict_user_defined_variables: false</code> set <code>ci_pipeline_variables_minimum_override_role</code> as <code>developer</code>.</p><h2 id=namespace-parameter-in-project-import-api-endpoints><code>namespace</code> parameter in project import API endpoints</h2><p>Breaking change. <a href=https://gitlab.com/gitlab-org/gitlab/-/issues/511053>Related issue</a>.</p><p>In GitLab 18.7, the <code>namespace</code> parameter in the <a href=/api/project_import_export/>project import and export API</a> is deprecated
in favor of the <code>namespace_id</code> and <code>namespace_path</code> parameters. The <code>namespace</code> parameter accepted both an ID or path,
which caused ambiguity when namespace paths contained only digits.</p><p>Instead, you should use:</p><ul><li><code>namespace_id</code> when specifying a namespace by its numeric ID.</li><li><code>namespace_path</code> when specifying a namespace by its path.</li></ul><p>The <code>namespace</code> parameter will be removed in v5 of the GitLab REST API.</p></div><div
